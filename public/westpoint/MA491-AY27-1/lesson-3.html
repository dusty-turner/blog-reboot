Lesson 3: Git & GitHub

Objectives

NoteBy the end of this lesson, you will be able to:
  • Explain why version control matters for data science work
  • Use core Git commands: git init, git add, git commit, git log, git diff
  • Write a .gitignore file to exclude files from tracking
  • Create a GitHub account and repository
  • Connect a local Git repo to GitHub and push commits
  • Describe branches and pull requests at a conceptual level

Before Class

  • Make sure you can SSH into your Pi and access RStudio Server
  • If you already have a GitHub account, have your credentials ready
  • If you don’t have one, go to github.com and create a free account before class

Lesson Outline

1. The Problem Version Control Solves (5 min)

You’ve done this:

report_v1.docx
report_v2.docx
report_v2_final.docx
report_v2_final_FINAL.docx
report_v2_final_FINAL_really_this_time.docx

Or worse — you changed a script, it broke, and you have no idea what it looked like before.

Git solves this. It tracks every change to every file. You can see exactly what changed, when, and why. You can go back to any previous version. You can work on new features without breaking what already works.

Every professional software team, data science team, and research lab uses version control. Starting today, so do you.

NoteGit vs. GitHub

Git is the version control tool that runs on your computer. GitHub is a website that hosts Git repositories online so you can back up your work, share it, and collaborate. You need Git. GitHub makes Git much more useful.

2. Git Concepts (10 min)

Four things to understand before we start typing commands:

Repository (repo): A directory that Git is tracking. Git stores a complete history of every change inside a hidden .git/ folder.

Staging area: A holding zone where you place changes before committing them. This lets you choose exactly which changes go into each commit.

Commit: A permanent snapshot of your staged changes. Think of it as a save point in a video game — you can always come back to it.

Branch: A parallel line of development. The default branch is called main. You can create branches to experiment without affecting main.

┌──────────────┐     git add     ┌──────────────┐    git commit    ┌──────────────┐
│   Working    │ ──────────────> │   Staging    │ ───────────────> │  Repository  │
│  Directory   │                 │    Area      │                  │   (History)  │
│              │                 │              │                  │              │
│ Your files   │                 │ "Ready to    │                  │ Permanent    │
│ as you edit  │                 │  commit"     │                  │ snapshots    │
└──────────────┘                 └──────────────┘                  └──────────────┘

3. Hands-On: Core Git Commands (20 min)

SSH into your Pi (or use the RStudio terminal). We’ll work in your ~/ma491 directory.

Install and Configure Git

sudo apt install -y git

git config --global user.name "Your Name"
git config --global user.email "your.email@westpoint.edu"
git config --global init.defaultBranch main

Initialize a Repository

cd ~/ma491
git init

You should see: Initialized empty Git repository in /home/pi/ma491/.git/

The Core Workflow: status, add, commit, log, diff

# Check what Git sees
git status

# Create a file
echo "# MA491 Data Science Practicum" > README.md

# Check status — README.md is "untracked"
git status

# Stage the file
git add README.md

# Check status — now it's under "Changes to be committed"
git status

# Commit with a message
git commit -m "Add README with course title"

# View the commit history
git log --oneline

Now make a change and see git diff in action:

# Edit the file
echo "" >> README.md
echo "Built on Raspberry Pi. Powered by AI." >> README.md

# See what changed (red = removed, green = added)
git diff

# Stage and commit
git add README.md
git commit -m "Add project tagline to README"

# Two commits now
git log --oneline
WarningWrite Good Commit Messages

Bad: "updated stuff", "fix", "asdfgh"

Good: "Add README with course title", "Fix column name typo in cleaning script"

A commit message should describe what changed and why. Your future self will thank you.

Practice Round

Do this on your own:

# Create a script file
echo '#!/bin/bash' > scripts/hello.sh
echo 'echo "Hello from MA491"' >> scripts/hello.sh
chmod +x scripts/hello.sh

# Stage, commit, verify
git add scripts/hello.sh
git commit -m "Add hello script"
git log --oneline       # Should show 3 commits

4. The .gitignore File (5 min)

Some files should never be tracked:

  • Secrets — API keys, passwords, .env files
  • Large data — Git is not built for big binary files
  • Temporary files — .RData, .Rhistory, cache folders
  • System junk — .DS_Store, Thumbs.db

Create a .gitignore:

nano .gitignore

Add these lines:

# R artifacts
.Rhistory
.RData
.Rproj.user/

# Data (track code, not data)
*.csv
*.xlsx
data/

# System files
.DS_Store
Thumbs.db

# Secrets
.env
*_secret*
*_key*

Save (Ctrl+O, Enter) and exit (Ctrl+X).

git add .gitignore
git commit -m "Add .gitignore for R artifacts, data, and secrets"
WarningNever Commit Secrets

If you accidentally commit a file with an API key, that key is in the Git history forever. Removing it requires rewriting history, which is messy. Prevention is much easier than cleanup. When in doubt, add it to .gitignore.

5. GitHub: Taking It Online (20 min)

Git on your Pi gives you version control locally. GitHub puts your repo in the cloud so you have a backup, a portfolio, and a way to collaborate.

Create a GitHub Account

If you haven’t already, go to github.com and create a free account. Use your westpoint.edu email — you get GitHub Pro for free with a student/educator account.

Create a Repository on GitHub

  1. Click the + in the top-right corner, then New repository
  2. Name it ma491 (match your local directory name)
  3. Leave it public (we want to build your portfolio)
  4. Do NOT check “Add a README” — you already have one locally
  5. Click Create repository

GitHub will show you a set of commands. Use the ones for “push an existing repository”:

Connect Local to Remote and Push

# Add GitHub as a remote named "origin"
git remote add origin https://github.com/YOUR_USERNAME/ma491.git

# Push your main branch to GitHub
git push -u origin main

You’ll be prompted for your GitHub username and password.

NoteAuthentication

GitHub no longer accepts passwords for command-line access. You’ll need a Personal Access Token (PAT):

  1. Go to GitHub > Settings > Developer settings > Personal access tokens > Tokens (classic)
  2. Generate a new token with repo scope
  3. Use this token as your password when prompted

Alternatively, ask your AI assistant: “How do I set up GitHub authentication with a personal access token on Linux?”

After pushing, refresh your GitHub repository page in the browser. Your files and commit history should be there.

Make Another Change and Push

echo "" >> README.md
echo "## Lessons" >> README.md
echo "- Lesson 1: Unboxing the Pi" >> README.md
echo "- Lesson 2: Linux, SSH, RStudio" >> README.md
echo "- Lesson 3: Git & GitHub" >> README.md

git add README.md
git commit -m "Add lesson list to README"
git push

Refresh GitHub — your new commit appears.

6. Branches and Pull Requests (10 min)

This section is conceptual with a brief demo. You’ll use branches more as the course progresses.

What Is a Branch?

A branch is a parallel copy of your code. You work on a branch without affecting main. When your work is ready, you merge it back.

main:       A --- B --- C
                   \
feature:            D --- E

Commits D and E happen on the feature branch. main is untouched until you merge.

Quick Demo

# Create and switch to a new branch
git checkout -b add-notes

# Make a change
echo "This branch is for experimentation." > notes.txt
git add notes.txt
git commit -m "Add notes file on feature branch"

# Switch back to main
git checkout main

# notes.txt doesn't exist here — it's only on the add-notes branch
ls notes.txt    # No such file

# Merge the branch into main
git merge add-notes

# Now notes.txt is on main too
ls notes.txt    # It's here now
git push

Pull Requests (Conceptual)

On GitHub, a pull request (PR) is a proposal to merge one branch into another. It lets teammates review your changes before they go into main. PRs are the standard workflow for professional teams. You’ll use them in group projects later this semester.

TipAI Assist

Git can feel confusing at first. When you hit a situation you don’t understand — a merge conflict, a detached HEAD, a rejected push — paste the error message into your AI assistant. It will explain what happened and how to fix it. This is one of the best uses of AI in a development workflow.

Quick Reference

Command What It Does
git init Initialize a new repository
git status Show what’s changed
git add <file> Stage a file for the next commit
git add . Stage everything
git commit -m "msg" Commit staged changes with a message
git log --oneline Show commit history (compact)
git diff Show unstaged changes
git remote add origin URL Connect to a remote repository
git push Push commits to the remote
git checkout -b name Create and switch to a new branch
git merge name Merge a branch into the current branch
