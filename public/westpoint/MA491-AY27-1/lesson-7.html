Lessons 7–10: Guided Project — NYC 311

Project Overview

NYC 311 is New York City’s non-emergency services hotline. Residents call or submit online complaints about noise, potholes, broken streetlights, rodents, illegal parking — anything the city should fix but hasn’t. The system logs millions of complaints per year, and the data is publicly available through the NYC Open Data portal.

Over the next four lessons, you will pull this data from the API, clean it, explore it, model it, and build an interactive dashboard that you deploy and host. This is a guided project — I will walk you through the major decisions and you will execute. The independent projects later in the course will give you less scaffolding.

What you will build: A hosted interactive dashboard that explores 311 complaint patterns across New York City.

What you will practice:

  • Working with a real API
  • Cleaning messy, real-world data
  • Exploratory data analysis and visualization
  • Fitting and evaluating a model
  • Building and deploying an interactive dashboard
  • Using Git at every step

AI usage: Use AI throughout this block. Claude is particularly useful for parsing API documentation, writing data cleaning code, and debugging deployment issues. As always, you must understand and be able to explain every line of code you submit.

The Data Source

NYC 311 data lives on the NYC Open Data portal and is accessible through the Socrata Open Data API (SODA). You do not need an API key for basic queries, but you will hit rate limits faster without one.

TipUse Claude here

The Socrata API documentation is dense. Ask Claude: “How do I query the NYC 311 dataset using the Socrata API in R with httr2? I want complaints from the last 12 months, limited to 50,000 rows.” It will generate the query string and explain the parameters.

Set Up the Project

# Create a new RStudio project (or a new directory in your existing repo)
# Project structure:
# nyc311/
#   data/          # cleaned data lives here
#   scripts/       # R scripts
#   dashboard/     # Quarto dashboard files (Lesson 10)
#   README.md

library(tidyverse)
library(httr2)
library(janitor)

Pull Data from the API

# Socrata API endpoint for NYC 311
base_url <- "https://data.cityofnewyork.us/resource/erm2-nwe9.json"

# Pull recent complaints — $where filters, $limit controls row count
resp <- request(base_url) |>
  req_url_query(
    `$where` = "created_date > '2025-01-01T00:00:00'",
    `$limit` = 50000,
    `$order` = "created_date DESC"
  ) |>
  req_perform()

# Parse the JSON response into a data frame
raw_311 <- resp |>
  resp_body_json(simplifyVector = TRUE) |>
  as_tibble()

glimpse(raw_311)
NoteAPI Pagination

The Socrata API returns a maximum of 50,000 rows per request by default. If you need more, use the $offset parameter to paginate. For this project, 50,000 rows is plenty to start.

Clean the Data

Real data is messy. This dataset will have wrong data types, missing values, and categories you do not need.

clean_311 <- raw_311 |>
  clean_names() |>                                    # janitor: snake_case column names
  mutate(
    created_date = as.POSIXct(created_date),          # fix date columns
    closed_date = as.POSIXct(closed_date),
    created_day = as.Date(created_date),              # extract just the date
    created_hour = hour(created_date),                # extract the hour
    response_hours = as.numeric(
      difftime(closed_date, created_date, units = "hours")
    )
  ) |>
  filter(
    borough != "Unspecified",                          # drop rows with no borough
    !is.na(complaint_type)                             # drop rows with no complaint type
  ) |>
  select(
    unique_key, created_date, closed_date, created_day, created_hour,
    response_hours, complaint_type, descriptor, borough,
    city, incident_zip, latitude, longitude, status
  )

glimpse(clean_311)
summary(clean_311)
WarningCheck Your Work

After cleaning, run summary() and look at the min/max of your date columns, the distribution of response_hours (are there negative values? huge outliers?), and the count of NAs in each column. Cleaning is never one pass — you will come back to this.

Save and Commit

# Save the cleaned data locally so you don't re-download every time
write_csv(clean_311, "data/clean_311.csv")

Fit a Model

The point here is not to build a great model. It is to go through the motions: pick a target, choose features, fit, evaluate. I will guide the choice in class, but here are three reasonable options:

Option A: Predict complaint type (classification)

library(tidymodels)

# Simplify: top 5 complaint types only
top_types <- clean_311 |>
  count(complaint_type, sort = TRUE) |>
  head(5) |>
  pull(complaint_type)

model_data <- clean_311 |>
  filter(complaint_type %in% top_types) |>
  mutate(complaint_type = factor(complaint_type)) |>
  select(complaint_type, borough, created_hour, latitude, longitude) |>
  drop_na()

# Split
set.seed(491)
split <- initial_split(model_data, prop = 0.8, strata = complaint_type)
train <- training(split)
test <- testing(split)

# Fit a simple model
fit <- multinom_reg() |>
  set_engine("nnet") |>
  fit(complaint_type ~ borough + created_hour, data = train)

# Evaluate
predictions <- predict(fit, test) |>
  bind_cols(test)

accuracy(predictions, truth = complaint_type, estimate = .pred_class)

Option B: Predict response time (regression)

model_data <- clean_311 |>
  filter(response_hours > 0, response_hours < 720) |>  # drop negatives and >30 days
  select(response_hours, complaint_type, borough, created_hour) |>
  drop_na()

fit <- linear_reg() |>
  set_engine("lm") |>
  fit(response_hours ~ complaint_type + borough + created_hour, data = model_data)

tidy(fit)
glance(fit)

Option C: Predict daily complaint volume (time series-ish regression)

daily <- clean_311 |>
  count(created_day) |>
  mutate(
    day_of_week = wday(created_day, label = TRUE),
    month = month(created_day, label = TRUE)
  )

fit <- linear_reg() |>
  set_engine("lm") |>
  fit(n ~ day_of_week + month, data = daily)

tidy(fit)
NoteModel Evaluation — Keep It Simple

For classification, report accuracy and compare to the baseline (most common class). For regression, report R-squared and RMSE. The goal is not a publication-quality model — it is understanding the end-to-end workflow.

Focus on the Story

By now you have cleaned data, several visualizations, and a model. Step back and ask: What is the one-sentence takeaway?

Some possibilities:

  • “Noise complaints dominate at night; heating complaints dominate in winter — the city’s problems follow predictable rhythms.”
  • “Response times vary dramatically by complaint type, and borough matters more than time of day.”
  • “You can predict the complaint type from just the location and time with X% accuracy, which tells us neighborhoods have persistent, characteristic problems.”

Render and Test Locally

quarto render index.qmd

Open the resulting HTML file in a browser. Does everything look right? Fix layout issues now.

Deploy

Option A: GitHub Pages (public)

# From the project root
quarto publish gh-pages dashboard/index.qmd

This will create a gh-pages branch and push the rendered site. Go to your repo settings on GitHub, enable Pages from the gh-pages branch, and your dashboard will be live at https://<username>.github.io/<repo>/.

Option B: Tailscale (private, accessible only on your Tailscale network)

If you set up Tailscale hosting in Lesson 6, copy the rendered HTML to your Pi’s served directory.

NoteDeployment Troubleshooting

Deployment is where things break. Common issues:

  • File paths: Your dashboard reads ../data/clean_311.csv — does that path exist in the deployed environment? You may need to embed the data or adjust paths.
  • Missing packages: GitHub Pages renders use your local machine. Make sure you render locally before publishing.
  • Large files: If your data file is too big for GitHub, filter to a smaller subset or use write_rds() for compression.

Use Claude to debug deployment errors — paste the full error message and your project structure.
