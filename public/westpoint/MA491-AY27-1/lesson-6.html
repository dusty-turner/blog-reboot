Lesson 6: Hosting — Local, Tailscale, & GitHub Pages

Overview

You built a dashboard. Right now it only works when you run it on your Pi and access it from the same network. Today you learn three levels of hosting — local, Tailscale, and GitHub Pages — so your work is accessible to others. By the end of this lesson, a classmate will be able to view your dashboard from their own device.

Objectives

NoteBy the end of this lesson, you will be able to:
  • Run a Shiny app on your Pi so it’s accessible on your local network
  • Install and configure Tailscale for secure remote access to your Pi
  • Render a static Quarto site and deploy it to GitHub Pages
  • Explain the trade-offs between local, Tailscale, and GitHub Pages hosting
  • Have at least one project accessible on each platform

Before Class

  • Have your Lesson 5 Shiny dashboard working on your Pi
  • Have a GitHub account and your ma491 repo set up (from Lesson 4)
  • Make sure Quarto is installed on your Pi:
# Check if Quarto is installed
quarto --version

# If not, install it (ARM64 version)
# Your instructor will provide the current download link

The Three Levels of Hosting

Level Access Dynamic? Public? Best For
Local Same Wi-Fi network Yes (Shiny runs live) No Development, classroom demos
Tailscale Anywhere on your Tailscale network Yes (Shiny runs live) No Sharing with team, remote work
GitHub Pages Anyone on the internet No (static HTML only) Yes Portfolios, reports, documentation

Level 1: Local Hosting (15 min)

You’ve actually been doing this already. When you run a Shiny app on your Pi and access it from your laptop, that’s local hosting. Let’s make it more robust.

Run Shiny on a fixed port

By default, Shiny picks a random port. To make it predictable:

# Run your dashboard on port 3838 (the Shiny convention)
shiny::runApp("~/ma491/scripts/lesson5_dashboard.R",
              host = "0.0.0.0",
              port = 3838)

Setting host = "0.0.0.0" means the app listens on all network interfaces, not just localhost. Anyone on your Wi-Fi can now access it.

Access from another device

On any device connected to the same network, open a browser and go to:

http://<your-pi-ip>:3838
TipFind Your Pi’s IP 
hostname -I

The first address (e.g., 192.168.1.42) is what you need.

Test it

Have the person sitting next to you open your dashboard in their browser. If they can see it, local hosting works.

Limitations

  • Only works on the same Wi-Fi network
  • If the Pi loses power or reboots, the app stops
  • Your IP address might change if DHCP reassigns it
NoteRunning in the Background

To keep the app running after you close your SSH session, you can use tmux or nohup:

# Option 1: tmux (recommended — you can reattach later)
tmux new -s dashboard
# Then run R and start the app
# Detach with Ctrl+B then D

# Option 2: nohup (simpler, no reattach)
nohup Rscript -e "shiny::runApp('~/ma491/scripts/lesson5_dashboard.R', host='0.0.0.0', port=3838)" &

Level 2: Tailscale (25 min)

Tailscale creates a private network (called a “tailnet”) between your devices. Once your Pi and your laptop are both on Tailscale, you can access your Pi from anywhere — your dorm, the library, off-post — as if you were on the same local network.

What is Tailscale?

  • A mesh VPN — encrypts traffic between your devices
  • No port forwarding — works through firewalls and NATs
  • Free tier — up to 100 devices, plenty for personal use
  • Each device gets a stable IP address (100.x.x.x) that never changes

Step 1: Create a Tailscale account

Go to tailscale.com and sign up. Use your personal email (not your @westpoint.edu — you want this to persist after graduation).

Step 2: Install Tailscale on your Pi

# Install Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# Start Tailscale and authenticate
sudo tailscale up

It will print a URL. Open that URL in your browser, sign in, and authorize your Pi.

Step 3: Install Tailscale on your laptop

Download and install Tailscale from tailscale.com/download. Sign in with the same account.

Step 4: Verify connectivity

# On your Pi, find its Tailscale IP
tailscale ip -4
# Something like 100.64.0.5

From your laptop, try pinging that IP:

ping 100.64.0.5

Step 5: Access your dashboard via Tailscale

Start your Shiny app on the Pi (same as before):

shiny::runApp("~/ma491/scripts/lesson5_dashboard.R",
              host = "0.0.0.0",
              port = 3838)

Now on your laptop (even from off-post), open:

http://100.64.0.5:3838
WarningTailscale IP vs. Local IP

Your Pi now has two IP addresses:

  • Local: 192.168.x.x — only works on the same Wi-Fi
  • Tailscale: 100.x.x.x — works from anywhere your laptop has internet

Use the Tailscale IP for remote access.

Step 6: Access RStudio Server via Tailscale too

Since Tailscale tunnels all traffic, your RStudio Server is also accessible remotely:

http://100.64.0.5:8787

Now you can code on your Pi from anywhere.

TipMagic DNS

Tailscale assigns human-readable names to your devices. Instead of 100.64.0.5, you might be able to use raspberry-pi.tailnet-name.ts.net. Check the Tailscale admin console for your device names.

Classmate verification

Pair up with a classmate. Add each other’s devices to a shared Tailscale network (or have your classmate install Tailscale and join your tailnet via an invite). Verify they can access your dashboard at your Tailscale IP.

Level 3: GitHub Pages (25 min)

GitHub Pages hosts static websites for free directly from a GitHub repository. It’s perfect for Quarto-rendered reports, slides, and documentation — but it cannot run a live Shiny app (no R server on GitHub).

What can you host on GitHub Pages?

  • Quarto-rendered HTML documents and websites
  • Static dashboards (Quarto dashboards with no server-side interactivity)
  • Reports, portfolios, project documentation
  • Observable JS plots (client-side interactivity without a server)

Step 1: Create a simple Quarto document

Create a file ~/ma491/report.qmd:

---
title: "MA491 Practicum Report"
author: "Your Name"
date: today
format: html
---

Step 2: Render to HTML

cd ~/ma491
quarto render report.qmd

This creates report.html in the same directory. Open it in your browser to verify it looks right.

Step 3: Set up GitHub Pages

You need to tell GitHub which folder to serve. The simplest approach: put your rendered HTML in a docs/ folder.

# Create the docs directory
mkdir -p ~/ma491/docs

# Render directly to docs/
quarto render report.qmd --output-dir docs

Or, for a full Quarto website, create a _quarto.yml:

project:
  type: website
  output-dir: docs

website:
  title: "MA491 Practicum"
  navbar:
    left:
      - href: index.qmd
        text: Home
      - href: report.qmd
        text: Report

format:
  html:
    theme: cosmo

Then render:

cd ~/ma491
quarto render

Step 4: Push and enable GitHub Pages

cd ~/ma491
git add docs/ _quarto.yml report.qmd
git commit -m "Add Quarto site for GitHub Pages"
git push

On GitHub:

  1. Go to your repo’s Settings tab
  2. Click Pages in the left sidebar
  3. Under “Source”, select Deploy from a branch
  4. Set the branch to main and the folder to /docs
  5. Click Save

Wait 1-2 minutes, then visit:

https://YOUR-USERNAME.github.io/ma491/
TipCustom Domain (Optional)

You can point a custom domain at your GitHub Pages site. This is optional but makes your portfolio look more professional. See GitHub’s documentation on custom domains.

Trade-offs Summary

Local Tailscale GitHub Pages
Access Same network only Anywhere (your devices) Public internet
Dynamic content Yes (Shiny, live data) Yes (Shiny, live data) No (static HTML)
Privacy Private Private Public
Cost Free Free (personal) Free
Uptime Only when Pi is on Only when Pi is on Always (GitHub hosts it)
Setup difficulty Easy Moderate Moderate
Best for Development Remote work, team sharing Portfolios, reports
NoteThe Right Tool for the Job

Use local hosting for development and quick demos. Use Tailscale when you need to access your Pi remotely or share a live dashboard with a small team. Use GitHub Pages when you want the world to see your polished, static work.

For a real production Shiny app that needs to be public and always-on, you’d use a service like shinyapps.io, Posit Connect, or a cloud VM. That’s beyond our scope but good to know about.
