Lessons 16–19: Connect Four

Project Overview

Build a fully playable Connect Four game with an AI opponent, then pit your bot against everyone else’s in a class tournament.

  • Game engine in R: board representation, move validation, win/draw detection
  • Shiny UI: clickable columns, colored pieces, real-time game state
  • AI opponent: start with a random bot, then make it smart with minimax and heuristics
  • Tournament: round-robin where every bot plays every other bot
  • AI assist throughout: use Claude to accelerate development — ask it to generate board logic, debug Shiny reactivity, explain minimax, and review your evaluation function

This is a four-lesson arc. By Lesson 19, you’ll have a hosted game that classmates can play against, and a tournament dashboard showing how your bot stacks up.

Lesson 16: Build the Game & Basic Bot

Goal: A working Connect Four game you can play against a random bot in a Shiny app.

75-Minute Outline

Time Activity
0–5 Intro: what we’re building over four lessons, show a finished example
5–20 Build the game engine together: board as a 6x7 matrix, drop_piece(), check_winner(), get_legal_moves(), is_draw()
20–35 Wire it into Shiny: clickable column buttons, render the board with colored circles, display whose turn it is
35–50 Build the random bot: pick a legal column at random, plug it into the game loop
50–65 Play against your random bot. It’s terrible — that’s the point. Note where it fails.
65–75 Wrap-up: what would make this bot smarter? Brainstorm strategies for Lesson 17

Key Concepts

  • Board representation: a matrix where 0 = empty, 1 = player, 2 = bot. Pieces drop to the lowest open row in a column.
  • Win detection: check all horizontal, vertical, and diagonal groups of four. A helper function that scans the board is the cleanest approach.
  • Shiny reactivity: the board state is a reactiveVal. Each move updates it, which triggers a re-render.

AI Assist

Use Claude to help you:

  • Generate the check_winner() function — describe what Connect Four winning looks like and let it write the scan logic
  • Debug Shiny reactivity issues (the board doesn’t update? paste your server code and ask)
  • Write a print_board() function for console testing before you have the UI working

Lesson 17: Make the Bot Smarter

Goal: Replace the random bot with one that actually thinks ahead using minimax and heuristics.

75-Minute Outline

Time Activity
0–5 Review: where did the random bot fail? What patterns did you notice?
5–15 Heuristic strategies: always take a winning move, always block an opponent’s winning move, prefer the center column
15–30 Minimax explained: the bot imagines future moves, assumes the opponent plays optimally, and picks the move that leads to the best worst-case outcome
30–40 Implement minimax: recursive function with a depth limit
40–50 Alpha-beta pruning: skip branches you already know are worse than what you’ve found. Same result, much faster.
50–65 Tune search depth as a difficulty knob. Depth 1 = easy. Depth 5+ = hard to beat. Test different depths.
65–75 Play against your improved bot. Can you still win?

Key Concepts

  • Heuristic evaluation: when you hit your depth limit, you need a function that scores the board. Simple version: count how many 3-in-a-rows each player has, weight center control, penalize giving the opponent a winning setup.
  • Minimax: the bot picks the move that maximizes its minimum guaranteed score. It assumes you’ll play perfectly against it.
  • Alpha-beta pruning: if you already found a move that guarantees a score of 5, and a new branch can’t possibly beat 5, stop exploring it. This lets you search deeper in the same time.
  • Depth as difficulty: depth 1 = only looks one move ahead (beatable). Depth 4–6 = looks several moves ahead (tough). Depth 7+ = very hard to beat but may be slow without pruning.

AI Assist

Use Claude to help you:

  • Explain minimax with a small game tree example before you implement it
  • Write the recursive minimax function — describe your board representation and scoring, and let it generate the skeleton
  • Debug alpha-beta pruning (it’s easy to get the bounds wrong)
  • Brainstorm evaluation function features: what makes a Connect Four position strong?

Lesson 18: Train & Improve

Goal: Find weaknesses in your bot, improve the evaluation function, and stress-test it with automated self-play.

75-Minute Outline

Time Activity
0–5 Review: how good is your bot? Where does it still lose?
5–20 Analyze losses: play against your bot deliberately and note what positions it misvalues. Is it missing traps? Ignoring threats?
20–35 Improve the evaluation function: add features like fork detection (two ways to win), threat counting, positional weights
35–50 Bot vs. bot: write a function that plays your bot against itself (or against a baseline bot) for 100+ games. Log results.
50–65 Set up overnight self-play on the Pi. The Pi runs 24/7 — let it grind through thousands of games while you sleep. Use the results to tune weights.
65–75 Compare before/after: is the new bot better? How do you measure “better”?

Key Concepts

  • Evaluation function tuning: the evaluation function is where the real intelligence lives. Minimax just searches; the evaluation function judges positions. Better judgment = better play.
  • Fork detection: a fork is when you create two winning threats at once. The opponent can only block one. This is the main pattern strong players exploit.
  • Self-play testing: have the bot play against a previous version of itself. If the new version wins >55% of games over hundreds of matches, it’s probably better.
  • The Pi as a compute resource: your Raspberry Pi is always on. Write a script that runs N games, logs results to a CSV, and saves it. Check the results in the morning.

AI Assist

Use Claude to help you:

  • Identify patterns in lost games — paste the board state where things went wrong and ask what the bot should have seen
  • Generate a self-play harness that logs game results
  • Suggest evaluation function features you haven’t thought of
  • Write the script that runs on the Pi overnight

Lesson 19: Tournament & Dashboard

Goal: Round-robin tournament. Every bot plays every other bot. Build a dashboard to display results. Host your game so classmates can play it.

75-Minute Outline

Time Activity
0–5 Intro: tournament rules. Each bot plays every other bot twice (once as Player 1, once as Player 2).
5–15 Standardize the bot interface: everyone’s bot must be a function that takes a board matrix and returns a column number. Package your bot as a single .R file.
15–30 Run the tournament: instructor collects all bot files, runs round-robin, streams results live
30–45 Build a tournament dashboard in Shiny: leaderboard, win/loss/draw rates, head-to-head matrix
45–55 Play your own bot. Can you beat it? Challenge: beat the tournament winner.
55–65 Host your game: deploy the Shiny app so classmates can play against your bot from their browser
65–75 Debrief: what worked, what didn’t, what would you do differently with more time

Key Concepts

  • Standardized interface: for the tournament to work, every bot must accept the same inputs and return the same outputs. This is a real software engineering constraint — agree on the API.
  • Round-robin: every bot plays every other bot. With N bots, that’s N*(N-1) games (each pair plays twice, alternating who goes first). First-player advantage is real in Connect Four.
  • Dashboard design: leaderboard is the headline, but head-to-head results tell the real story. Did one bot consistently beat everyone except one specific opponent?

AI Assist

Use Claude to help you:

  • Write the round-robin tournament runner
  • Build the dashboard layout (leaderboard table, head-to-head heatmap, win-rate bar chart)
  • Debug deployment issues when hosting the Shiny app
  • Analyze tournament results — ask it to find interesting patterns in the data
