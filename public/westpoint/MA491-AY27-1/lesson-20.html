Lessons 20–24: Single-Player Game

Project Overview

Design and build an original single-player game with multiple levels and progressive difficulty. This is your most creative project of the semester — you pick the concept, you design the mechanics, you build the whole thing.

  • Original game: puzzle, strategy, reaction, trivia, maze, platformer, tower defense, typing challenge — whatever interests you
  • Multiple levels: at least 3 levels with increasing difficulty
  • Hosted and playable: classmates will play your game on Game Arcade day
  • AI-assisted development: use Claude throughout to accelerate building, debug tricky logic, and generate content (level designs, trivia questions, enemy patterns)

Lesson 24 is Game Arcade: all games are live, everyone plays each other’s games, and the class votes on favorites.

Lesson 20: Concept & Design

Goal: Decide what game you’re building and create a design document that’s specific enough to start coding next class.

75-Minute Outline

Time Activity
0–5 Intro: project scope, timeline, what “done” looks like
5–15 Inspiration round: look at examples of simple but engaging browser games. What makes them fun? What’s the core loop?
15–30 Brainstorm your game concept. Think about: what’s the core mechanic? What makes each level harder? What’s the win condition?
30–45 Write the game design document (see template below)
45–60 Wireframe the UI: sketch the main game screen, the level select screen, and the game-over screen. Paper or digital, doesn’t matter.
60–70 Share concepts with a partner. Give each other feedback: is the scope realistic? Is the difficulty progression clear?
70–75 Finalize your design doc. You’re coding next class.

Game Design Document Template

Fill this out before you write any code:

  1. Game title
  2. One-sentence description (e.g., “A typing speed game where words fall faster each level”)
  3. Core mechanic: what does the player do every second? Click? Type? Drag? Decide?
  4. Level progression: how do levels get harder? Faster speed? More enemies? Tighter time limits? New mechanics?
  5. Win condition: how does the player beat a level?
  6. Lose condition: how does the player fail?
  7. Minimum 3 levels described: what’s different about each one?
  8. Tech stack: Shiny? Plain HTML/JS? Something else?
  9. Biggest technical risk: what’s the hardest part to build?

Game Ideas for Inspiration

  • Memory match with a timer that shrinks each level
  • Maze generator where mazes get bigger and more complex
  • Trivia game where questions get harder and the timer gets shorter (use Claude to generate questions)
  • Reaction time challenge with increasingly fast targets
  • Tower defense with waves of enemies
  • Word scramble with longer words and less time
  • Snake with obstacles that appear in later levels
  • Math blitz where operations get more complex each level
  • Rhythm game where patterns get faster
  • Puzzle slider with larger grids each level

You are not limited to this list. Build something you’d actually want to play.

AI Assist

Use Claude to help you:

  • Brainstorm game ideas — describe what kinds of games you enjoy and ask for concepts that would work as browser games with levels
  • Flesh out your design doc — paste your rough idea and ask it to identify gaps (what’s the lose condition? how does difficulty scale?)
  • Generate level content (trivia questions, word lists, maze layouts, enemy wave patterns)
  • Evaluate scope: paste your design doc and ask “can a student build this in 3 class periods?”

Lesson 21: Build Core Mechanics & Levels

Goal: Get the core game loop working with at least 3 levels. Playable but not polished.

75-Minute Outline

Time Activity
0–5 Quick check: does everyone have a clear plan from the design doc? Last chance to adjust scope.
5–15 Set up project structure: create the Shiny app (or HTML/JS project), stub out the main screens
15–45 Build the core mechanic. This is the main work block. Focus on making one level work first.
45–55 Add level progression: implement at least 2 more levels with increasing difficulty
55–65 Test: play through all levels yourself. Does the difficulty curve feel right?
65–75 Status check: where are you? What’s left? Make a plan for next class.

Key Concepts

  • Core loop first: get the basic mechanic working before you add levels, scoring, or polish. If the core loop isn’t fun, nothing else matters.
  • Difficulty progression: the simplest approaches — faster speed, more items, less time, larger grids — are often the most effective. You don’t need a new mechanic every level.
  • State management: your game has state (score, current level, lives remaining, timer). Decide early how you’re tracking it. In Shiny, reactiveValues works well.
  • Keep it simple: a simple game that works is better than an ambitious game that’s broken. You can always add complexity later.

AI Assist

Use Claude to help you:

  • Generate boilerplate: “I’m building a Shiny app for a memory match game. Set up the basic structure with a game board, a timer, and a score display.”
  • Debug game logic: paste your code and describe what’s going wrong
  • Implement specific mechanics: “How do I make a countdown timer in Shiny that resets each level?”
  • Generate level content programmatically: “Write a function that generates a random maze of size N x N”

What to Have Done by End of Class

Lesson 22: Polish & Playtest

Goal: Make the game actually fun. Swap with a classmate, get real feedback, and fix what’s broken.

75-Minute Outline

Time Activity
0–5 Quick demos: show your partner what you’ve built so far
5–25 Playtest swap: play your partner’s game for 20 minutes. Take notes on what’s fun, what’s confusing, what’s broken, what’s frustrating. Be specific.
25–35 Exchange feedback. Discuss what you found. Prioritize: what are the top 3 things to fix?
35–55 Fix and polish based on feedback. Focus on the biggest issues first.
55–65 Add polish: score display, level transitions, game-over screen, sound effects (optional), animations, instructions for new players
65–75 Second playtest: have your partner try again. Did the fixes help?

Playtest Feedback Template

When playing your partner’s game, note:

  1. First impression: did you understand what to do without being told?
  2. Fun factor: what moment was the most fun? When were you bored?
  3. Difficulty curve: was Level 1 too easy? Too hard? Did difficulty increase smoothly?
  4. Bugs: anything broken? Unexpected behavior?
  5. UX issues: confusing controls? Missing feedback? Hard to tell what’s happening?
  6. One thing to add: if you could add one feature, what would it be?

AI Assist

Use Claude to help you:

  • Fix bugs fast: paste the error or describe the unexpected behavior
  • Add polish: “Add a smooth transition between levels” or “Make the game-over screen show the final score and a restart button”
  • Generate instructions: paste your game description and ask for clear player-facing instructions
  • Add sound or animation: “How do I play a sound effect when the player scores in Shiny?”

What to Have Done by End of Class

Lesson 23: Final Build & Hosting

Goal: Final polish, deploy the game, and add it to your portfolio.

75-Minute Outline

Time Activity
0–5 Final stretch: what’s left to fix? Make a short punch list.
5–30 Final development: last bug fixes, final polish, edge case handling
30–45 Host the game: deploy so it’s accessible from a URL. Options: Shiny Server on the Pi, GitHub Pages (for HTML/JS games), Netlify, or Tailscale for local hosting.
45–55 Test the hosted version: does it work on someone else’s machine? Different browser?
55–65 Add to your portfolio website: link to the game, add a screenshot, write a one-sentence description
65–75 Final test: have someone play the hosted version end-to-end. Confirm everything works for Game Arcade tomorrow.

Hosting Options

  • Shiny Server on the Pi: if your game is a Shiny app, deploy it to the class Shiny Server. Accessible on the local network.
  • GitHub Pages: if your game is HTML/CSS/JS, push it to a GitHub repo and enable Pages. Free, public URL.
  • Netlify: drag and drop your build folder. Free tier is plenty. Good for static sites.
  • Tailscale: if you want to host from your own machine without exposing it to the public internet.

Pick the option that fits your tech stack. The only requirement is: classmates can access it from a URL during Game Arcade.

AI Assist

Use Claude to help you:

  • Debug deployment issues: “My Shiny app works locally but not on the server”
  • Write deployment scripts or config files
  • Generate a portfolio entry: paste your game description and ask for a concise portfolio blurb
  • Last-minute bug fixes under time pressure

What to Have Done by End of Class

Lesson 24: Game Arcade

Goal: Play each other’s games. Celebrate what everyone built. Vote on favorites.

75-Minute Outline

Time Activity
0–5 Setup: everyone confirms their game is live and the URL works
5–10 Share URLs: post your game link in the class channel
10–55 Game Arcade: play as many games as you can. Spend at least 3–5 minutes on each. Fill out the voting form as you go.
55–65 Voting closes. Tally results.
65–75 Awards and debrief: announce winners, discuss what made the best games stand out, reflect on the project

Voting Categories

  • Best Overall Game: the complete package — fun, polished, good difficulty curve
  • Most Creative: the most original concept or unexpected twist
  • Most Addictive: the game you kept wanting to play “one more time”
  • Best Design: the most polished UI, best visual design, smoothest experience

Every student votes in every category (you can’t vote for yourself). Ties are broken by the instructor’s unimpeachable judgment.

What Makes a Game Arcade Great

  • Play everything: don’t just play your friends’ games. Try them all.
  • Give real feedback: if you loved something, say so. If something could be better, say that too.
  • Appreciate the range: some games will be simple and elegant. Some will be ambitious and rough. Both are valid.
  • Have fun: this is the payoff for three lessons of building. Enjoy it.
