Lesson 4: Getting Data — Databases & APIs

Overview

So far you’ve loaded data from flat files. But most real-world data lives in databases or behind APIs. Today you learn how to connect to both from R: query a relational database with SQL, and call a public API to pull live data. By the end of this lesson you’ll have one script that talks to a database and an API, returning tidy data frames from each.

Objectives

NoteBy the end of this lesson, you will be able to:
  • Explain what a relational database is and why it matters
  • Write basic SQL queries: SELECT, WHERE, ORDER BY, JOIN, GROUP BY
  • Connect to a SQLite database from R and run queries
  • Explain what an API is and how HTTP requests work
  • Call a public API from R and parse the JSON response into a data frame
  • Save both results as data frames in a single R script

install.packages(c("DBI", "RSQLite", "httr2", "jsonlite"))

Part 1: What Is a Database? (10 min)

A database is an organized collection of data stored in tables. Each table has:

  • Rows — individual records (one cadet, one transaction, one sensor reading)
  • Columns — attributes of each record (name, GPA, timestamp)

Tables can be related to each other through shared columns (keys). That’s what makes it a relational database.

┌─────────────────────┐       ┌─────────────────────┐
│  cadets             │       │  grades              │
├─────────────────────┤       ├─────────────────────┤
│ cadet_id  (PK)      │──────>│ cadet_id  (FK)       │
│ name                │       │ course               │
│ company             │       │ grade                │
│ class_year          │       │ semester             │
└─────────────────────┘       └─────────────────────┘
  • PK = Primary Key (unique identifier for each row)
  • FK = Foreign Key (links to a row in another table)
TipWhy Not Just Use CSV Files?

Databases handle millions of rows efficiently, enforce data integrity, support concurrent users, and let you query exactly the data you need instead of loading everything into memory.

Part 2: SQL Basics (15 min)

SQL (Structured Query Language) is how you talk to databases. You don’t need to memorize everything — you need to know the core verbs.

SELECT and WHERE

-- Get all columns from the cadets table
SELECT * FROM cadets;

-- Get specific columns
SELECT name, company FROM cadets;

-- Filter rows
SELECT name, gpa FROM cadets WHERE gpa > 3.5;

-- Combine conditions
SELECT name, gpa FROM cadets WHERE gpa > 3.0 AND company = 'A';

ORDER BY

-- Sort by GPA, highest first
SELECT name, gpa FROM cadets ORDER BY gpa DESC;

GROUP BY with Aggregates

-- Average GPA by company
SELECT company, AVG(gpa) as avg_gpa
FROM cadets
GROUP BY company;

-- Count cadets per class year
SELECT class_year, COUNT(*) as n
FROM cadets
GROUP BY class_year;

JOIN

-- Combine cadets and grades tables
SELECT c.name, g.course, g.grade
FROM cadets c
JOIN grades g ON c.cadet_id = g.cadet_id
WHERE g.course = 'MA491';
NoteSQL Is Everywhere

SQL is the most widely used data language in the world. It works the same whether you’re querying SQLite on a Raspberry Pi or a massive Postgres database at Amazon. Learn it once, use it everywhere.

Part 3: Connect to a Database from R (15 min)

We’ll use SQLite — a lightweight database stored as a single file. No server needed, perfect for your Pi.

Create a sample database

library(DBI)
library(RSQLite)

# Create (or connect to) a local database file
con <- dbConnect(SQLite(), "~/ma491/data/practicum.db")

# Create a cadets table
cadets <- data.frame(
  cadet_id = 1:8,
  name = c("Adams", "Baker", "Clark", "Davis",
           "Evans", "Foster", "Garcia", "Hill"),
  company = c("A", "B", "C", "A", "B", "C", "A", "B"),
  gpa = c(3.5, 3.2, 3.8, 2.9, 3.6, 3.1, 3.9, 2.7)
)

dbWriteTable(con, "cadets", cadets, overwrite = TRUE)

# Create a grades table
grades <- data.frame(
  cadet_id = c(1, 1, 2, 3, 4, 5, 6, 7, 8),
  course = c("MA491", "CS394", "MA491", "MA491", "CS394",
             "MA491", "MA491", "CS394", "MA491"),
  grade = c(92, 88, 85, 97, 79, 91, 83, 90, 76)
)

dbWriteTable(con, "grades", grades, overwrite = TRUE)

Query the database

# Run a SQL query and get back a data frame
result <- dbGetQuery(con, "SELECT * FROM cadets WHERE gpa > 3.0")
result

# Join tables
joined <- dbGetQuery(con, "
  SELECT c.name, c.company, g.course, g.grade
  FROM cadets c
  JOIN grades g ON c.cadet_id = g.cadet_id
  ORDER BY g.grade DESC
")
joined

# Aggregate
avg_by_company <- dbGetQuery(con, "
  SELECT company, AVG(gpa) as avg_gpa, COUNT(*) as n
  FROM cadets
  GROUP BY company
")
avg_by_company

# Always disconnect when done
dbDisconnect(con)
TipAI Assist

SQL syntax can be hard to remember at first. Use Claude or Copilot: describe what you want in plain English (“join these two tables on cadet_id and give me the average grade by company”) and let the AI write the SQL. Then read the query and make sure you understand what it does.

Part 4: What Is an API? (10 min)

An API (Application Programming Interface) is a way for programs to talk to each other over the internet. You send a request to a URL (called an endpoint), and you get back a response — usually in JSON format.

Your R Script                     API Server
     │                                │
     │── GET /api/weather?city=NYC ──>│
     │                                │
     │<── JSON response ──────────────│
     │  { "temp": 72, "wind": 8 }    │

Key concepts:

  • Endpoint — the URL you send a request to
  • HTTP Methods — GET (read data), POST (send data)
  • JSON — the data format most APIs return (key-value pairs, nested structures)
  • API Key — some APIs require a key for authentication (free tier usually available)

Part 5: Call an API from R (15 min)

We’ll use httr2 to call a free, no-auth API. The Open-Meteo weather API is a good example — no key required.

Make a request

library(httr2)
library(jsonlite)

# Build the request to Open-Meteo (West Point coordinates)
resp <- request("https://api.open-meteo.com/v1/forecast") |>
  req_url_query(
    latitude = 41.39,
    longitude = -73.95,
    hourly = "temperature_2m,wind_speed_10m",
    temperature_unit = "fahrenheit",
    forecast_days = 1
  ) |>
  req_perform()

# Check the status (200 = success)
resp_status(resp)

Parse the JSON response

# Extract JSON body as a list
weather_data <- resp |> resp_body_json()

# Look at the structure
str(weather_data, max.level = 2)

# Pull out the hourly data into a data frame
weather_df <- data.frame(
  time = unlist(weather_data$hourly$time),
  temp_f = unlist(weather_data$hourly$temperature_2m),
  wind_mph = unlist(weather_data$hourly$wind_speed_10m)
)

head(weather_df)

Quick visualization

library(ggplot2)

ggplot(weather_df, aes(x = seq_along(temp_f), y = temp_f)) +
  geom_line(color = "steelblue", linewidth = 1) +
  geom_point(color = "steelblue") +
  labs(
    title = "West Point Temperature Forecast (Next 24 Hours)",
    x = "Hour",
    y = "Temperature (°F)"
  ) +
  theme_minimal()
NoteOther Free APIs to Explore

  • USGS Earthquake Data: https://earthquake.usgs.gov/fdsnws/event/1/query
  • NASA APOD: https://api.nasa.gov/planetary/apod (free API key)
  • Dog CEO: https://dog.ceo/api/breeds/list/all (no key, good for quick testing)
